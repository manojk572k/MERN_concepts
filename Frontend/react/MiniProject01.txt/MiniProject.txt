Mini Project: User Directory App

This project is a small real-world React app where I:

fetch users from an API
display them on screen
search users by text
optimize search using debouncing
organize UI into components

It is a very good project because it combines almost everything already learned.

1. Main purpose of this project

The goal is to build a searchable user directory.

Think of it like:

company employee list
contact list
customer list
admin dashboard user panel

A user types in a search box, and matching users are shown.

So this project is not just “display data.”
It teaches how data moves through a React app.

3. High-level app flow

The full app flow is:

App loads
→ API request starts
→ loading message appears
→ data comes from API
→ users stored in state
→ users displayed on screen

User types in search box
→ search state updates
→ debounced value updates after delay
→ filtered users are calculated
→ only matching users appear

This is the full behavior of the app.

4. Main components in the project

A clean version of this app can have these components:

App

This is the main controller.

Responsibilities:

store all important state
fetch API data
handle loading and error
hold search value
use debounce logic
create filtered users
pass data to child components

This is the “brain” of the app.

SearchBox

This is only for the input UI.

Responsibilities:

show the search input
receive current search text from parent
call parent function when typing happens

This component should not own the main search state.
It only displays and sends user input upward.

UserList

This is responsible for showing a list of users.

Responsibilities:

receive filtered users as props
loop through them
render one UserCard for each user

This keeps list rendering separate from the app logic.

UserCard

This is for showing one single user.

Responsibilities:

display user information like name and email
later can also show phone, website, company, etc.

This is the single row/item UI.

5. Why split into these components?

Because each part has a separate responsibility.

Without splitting

Everything stays in one file:

fetch logic
input logic
filter logic
list logic
item UI

That becomes messy fast.

With splitting

Each component has one clear job:

App = logic
SearchBox = input UI
UserList = list UI
UserCard = single item UI

This is the beginning of real React architecture.

6. State needed in the project

The project mainly needs these states in App:

users

Stores API data.

This is the original full list from the server.

Example idea:

10 users fetched from API
stored in users
loading

Tells whether API request is still in progress.

Why useful:

before data comes, show “Loading...”
prevents blank/confusing UI
error

Stores any fetch failure message.

Why useful:

if internet/API fails, show message instead of broken UI
search

Stores whatever the user is typing right now.

Why useful:

controlled search input
UI always knows current typed value
debouncedSearch

Stores delayed/stable search value.

Why useful:

avoids immediate filtering or repeated API behavior on every keystroke
improves performance and user experience
7. Difference between search and debouncedSearch

This is very important.

search

Immediate value.

If user types:

r
re
rea

then search changes immediately every time.

debouncedSearch

Delayed value.

If debounce delay is 500ms:

user types quickly
app waits
after typing stops for 500ms, then updates

So:

search = fast changing
debouncedSearch = delayed stable value

This is the core of debounce behavior.

8. How filtering works in this project

Once users are fetched, I do not need to fetch again for basic search.

Instead:

keep original array in users
create a filtered array based on search text

Filtering logic idea:

compare typed text with user name
maybe also compare with email
ignore uppercase/lowercase differences

That is called client-side filtering.

9. Why case-insensitive search matters

Suppose a user name is:

Leanne Graham

If the user types:

leanne
LEANNE
LeAnNe

all of these should still work.

So converting both sides to lowercase makes search more user-friendly.

This is small, but very important in real UI.

10. Conditional rendering in this project

This project is excellent for understanding conditional rendering.

There are several UI states:

Loading state

When request is in progress:

show loading message
Error state

When request fails:

show error message
Empty state

When no users match the search:

show “No users found”
Success state

When data exists:

render user list

So the app does not always show the same UI.
It changes based on current state.

That is exactly why React is powerful.

13. Why state should stay in App

Because multiple components need shared data.

For example:

SearchBox affects the search
UserList depends on the search result
App controls fetch, loading, error

So the common parent should own shared state.

That is lifting state up in action.

If SearchBox owned search completely, App and UserList would not easily coordinate around it.

17. Common mistakes in this project

These are good to watch out for:

Filtering mistakes
forgetting to return in filter
filtering wrong field
not handling uppercase/lowercase
Debounce mistakes
forgetting clearTimeout(timer)
using immediate search instead of debounced value
API mistakes
forgetting await res.json()
not handling loading/error properly
Component mistakes
putting too much logic in small UI components
duplicating shared state in children

These are normal mistakes, and fixing them teaches a lot.
